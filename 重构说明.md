# 重构说明

本文档记录本次围绕“运行模式、工具状态、fallback 标记、external/surrogate 边界”的重构内容，并给出下一轮优化指导。

## 一、重构背景

之前项目表面上可以跑完整流程，但真实情况不够透明：

- 一些外部化学工具没有真正运行成功时，系统会使用内部估算或 fallback 继续往下跑。
- 用户很难从结果里判断某个 docking、ADMET、逆合成、生成结果到底来自真实工具，还是来自 surrogate/fallback。
- 多轮迭代时，不同轮次的评估、排序、自我反驳结果容易混在一起，后续做“第一轮/第二轮/第三轮”记录会不清楚。
- 工具状态检查、运行模式、结果证据等级没有形成统一边界，导致前端和报告层很容易误把替代结果当成外部工具结果。

这次重构的目标不是让所有外部工具立刻都变成最优生产级运行，而是先把系统的“证据来源”讲清楚：真实外部工具就是 external，内部估算就是 surrogate，工具失败后的兜底就是 fallback，排序这种规则计算就是 internal heuristic。

## 二、核心概念边界

### 1. external tool

指真实调用外部科学工具并获得有效结果，例如：

- GNINA / Vina / DiffDock docking
- ADMET-AI
- AiZynthFinder
- REINVENT4
- AutoGrow4
- CReM

只有工具实际执行并产出有效结果时，才应该标记为 external tool used。

### 2. surrogate

指系统内部用 RDKit 描述符、规则、轻量估算逻辑模拟一个科学判断。

surrogate 可以帮助流程不断掉，但它不是外部工具结果，不能当成真实 docking 或真实 ADMET 预测。

### 3. fallback

指系统本来想用外部工具，但外部工具不可用、失败、超时或输入不满足要求，于是退回到 surrogate 或保守默认逻辑。

fallback 的关键点是：用户请求过 external，但最终没有完全拿到 external 结果。

### 4. internal heuristic

指系统本来就不依赖外部工具，而是使用内部规则计算，例如候选分子综合排序。

排序可以使用 external/surrogate/fallback 证据作为输入，但排序行为本身属于 internal heuristic。

## 三、本次主要改动

### 1. 候选评估结果增加运行模式和证据来源

涉及文件：

- `src/medagent/services/candidate_assessment.py`
- `src/medagent/domain/schemas.py`
- `src/medagent/api/app.py`
- `tests/test_candidate_assessment.py`

新增或强化字段：

- `round_id`
- `execution_mode`
- `evidence_tier`
- `external_tool_requested`
- `external_tool_used`
- `surrogate_used`
- `fallback_used`
- `external_attempted_count`
- `external_success_count`
- `surrogate_count`
- `fallback_count`

现在候选评估会区分几种情况：

- `surrogate_only`：只使用内部估算。
- `external_tool`：外部工具成功产出结果。
- `surrogate_fallback`：请求外部工具，但失败后退回内部估算。
- `mixed_external_surrogate`：同一批评估中有一部分是真实工具，一部分是 surrogate/fallback。

这样用户看到结果时可以知道：这个分子的 docking/admet/合成路线到底是实测外部工具结果，还是系统估算出来的。

### 2. 排序结果按轮次隔离，并标记为 internal heuristic

涉及文件：

- `src/medagent/services/candidate_ranking.py`
- `src/medagent/domain/schemas.py`
- `src/medagent/api/app.py`
- `tests/test_candidate_ranking.py`

改动重点：

- `generate_project_rankings()` 支持 `round_id`。
- `list_project_rankings()` 支持按 `round_id` 查询。
- 重新生成排序时，只替换当前轮次的排序结果，避免覆盖其他轮次。
- 排序结果增加运行模式字段，明确排序本身是 `internal_heuristic`。

排序会读取 docking、ADMET、合成路线、自我反驳等证据，但排序本身不是外部科学工具，所以不能标成 external。

### 3. 生成阶段透出工具执行状态和 fallback 信息

涉及文件：

- `src/medagent/services/molecule_generation.py`

改动重点：

这样以后用户选择 REINVENT4 / AutoGrow4 / CReM 时，系统不会只说“生成了分子”，而会说明生成来源是否真实来自对应工具。

## 五、现在项目的运行解释应该怎么说

以后不要再简单说“项目跑通了”，而应该说：

1. 生成阶段是否真实调用了 CReM / REINVENT4 / AutoGrow4。
2. 评估阶段哪些分子用了真实外部工具，哪些分子用了 surrogate。
3. 是否发生 fallback，为什么 fallback。
4. 排序基于哪些证据，证据里 external 占多少，surrogate/fallback 占多少。
5. 当前结果属于第几轮。

一个更准确的结果描述应该类似：

> 本轮生成 30 个分子，其中 10 个来自 CReM external tool，20 个来自 REINVENT4 surrogate fallback；候选评估阶段 docking 对 top 5 调用了 Vina，其余分子使用 surrogate docking 估算；最终排序为 internal heuristic，证据置信度较低的分子已标记。

这种说法虽然没那么漂亮，但对科研软件更诚实。

## 六、下一次优化指导

### 1. 前端必须显示工具来源

下一轮优先改前端展示：

- 分子卡片显示 `external / surrogate / fallback` 标签。
- 候选评估表格显示每一项证据来源。
- 排序页显示 evidence confidence。
- 工具失败时不要只显示“失败”，要显示“已 fallback 到内部估算”。

建议使用四种固定标签：

- External
- Surrogate
- Fallback
- Internal

### 2. 工具状态页要从“能不能用”升级为“能不能用于生产”

现在工具状态检查还不够细。下一轮应该把工具状态分成：

- `available`：命令或 Docker 镜像存在。
- `configured`：模型文件、数据库、receptor、grid 等配置齐全。
- `gpu_available`：GPU 容器可用。
- `smoke_test_passed`：最小样例跑通。
- `production_ready`：可以被真实项目调用。

例如 DiffDock 不能只看 Docker 镜像是否存在，还要检查：

- score model 是否存在。
- confidence model 是否存在。
- ESM2 缓存是否存在或可下载。
- NVIDIA Docker 是否可用。
- receptor/ligand 输入格式是否满足要求。

### 3. 评估流程要明确“粗筛”和“精筛”

建议把候选评估分成两层：

1. 粗筛：全部分子走 surrogate，速度快。
2. 精筛：只对 top N 调真实外部工具。

默认策略可以是：

- 50 个以内分子：top 10 外部工具精筛。
- 50-200 个分子：top 10-20 外部工具精筛。
- 200 个以上分子：先 surrogate 粗筛，再分批外部精筛。

不要默认所有分子都跑 DiffDock/GNINA/AiZynthFinder，否则计算量会炸。

### 4. REINVENT4 和 AutoGrow4 要进入“方法特异流程”

下一次核心流程优化应把三种生成方法拆开：

- CReM：以 seed 为中心做局部扩散，适合快速结构邻域探索。
- REINVENT4：适合 reward-driven RL，重点是奖励函数和目标约束。
- AutoGrow4：适合遗传算法搜索，重点是 receptor、binding pocket、docking/scoring 配置。

不要再让三者共享同一套迭代轮数。

建议下一轮实现：

- CReM 由用户设置每个 seed 生成几个分子。
- REINVENT4 由用户设置 reward profile、RL steps、采样数量。
- AutoGrow4 由用户设置遗传代数、种群规模、docking 配置。

### 5. SAR 桥接要真正进入生成约束

现在 SAR 桥接只是骨架。下一轮要把 SAR 输出转换成具体约束，例如：

- MW 建议范围。
- LogP 建议范围。
- TPSA 建议范围。
- HBD/HBA 上限。
- rotatable bonds 上限。
- 保留/避免的子结构。
- 与 seed 的相似度范围。
- docking pose 或关键相互作用建议。

自我反驳节点也应该输出更具体的下一轮优化建议，例如：

- “当前 top 分子 LogP 偏高，下一轮将 cLogP 目标压到 2.0-3.5。”
- “TPSA 过低可能影响溶解性，下一轮提高 TPSA 下限到 55。”
- “MW 接近 500，下一轮限制新增片段大小。”
- “HBD=0 且结合模式缺少氢键供体，下一轮优先引入弱供氢基团。”

### 6. 数据库轮次模型要继续补全

当前已开始传递 `round_id`，下一轮应继续完善：

- ProjectRound：记录每一轮输入条件、用户修改、开始/结束时间、状态。
- CampaignRun：记录每种生成方法的一次运行。
- 每轮 top candidates：记录哪些分子被用户选入下一轮。
- 每轮配置快照：保存当轮 reward、filter、docking grid、生成参数。

重点是：下一轮不是自动开始，而是用户基于上一轮结果手动改条件后开始。

### 7. 外部工具失败要留下可诊断日志

下一轮建议统一保存：

- command
- docker image
- mounted paths
- stdout tail
- stderr tail
- exit code
- timeout
- input file paths
- output file paths

这样用户再问“为什么卡住”时，可以直接定位是模型文件、GPU、输入格式、Docker 挂载还是工具本身失败。

## 七、建议的下一轮优先级

优先级 1：

- 前端/API 显示 external/surrogate/fallback 标签。
- 工具状态页增加 production_ready。
- 评估结果列表支持按 `round_id` 过滤并显示证据来源。

优先级 2：

- 完成 ProjectRound / CampaignRun 的实际业务闭环。
- 把 CReM、REINVENT4、AutoGrow4 拆成各自的运行配置。
- 让用户手动选择 top molecules 进入下一轮。

优先级 3：

- 完成 SAR 到生成约束的真实桥接。
- 强化自我反驳输出，让它给出具体理化性质调整建议。
- 为 REINVENT4 reward function 做可配置模板。

优先级 4：

- 对 DiffDock、AutoGrow4、REINVENT4 增加更完整的生产级 smoke test。
- 优化长耗时 integration test，把报告生成和核心编排逻辑拆开测。

## 八、验收标准

下一轮优化完成后，至少应满足：

- 用户能看到每个结果来自 external、surrogate、fallback 还是 internal。
- 每个项目可以按第一轮、第二轮、第三轮查看生成、评估、排序、自我反驳结果。
- 外部工具失败时，系统不会静默伪装成成功，而是明确写出 fallback 原因。
- 排序结果能说明自己基于哪些证据，以及证据置信度。
- 用户可以手动选择上一轮 top 分子进入下一轮。
- CReM、REINVENT4、AutoGrow4 不再强行共享同一套迭代参数。
